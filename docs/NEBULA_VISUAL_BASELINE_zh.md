# Nebula 视觉基准与动态适配说明

## 0. 文档目的

这份文档用于把已确认的 4 张 AI 视觉稿转成真正可执行的前端实现基准，避免后续开发时出现“只参考感觉、不参考细节”的漂移。

核心原则只有一句：

首页和节点详情页以视觉稿为主基准，默认按接近像素级还原；数据内容按当前 Komari 公开能力做动态适配。

## 2. 实现优先级规则

1. 桌面图优先级高于手机图，用于定义整体布局骨架、比例、卡片材质、间距系统、阴影和图表风格。
2. 手机图用于定义移动端层级、手势路径、底部导航、卡片堆叠和摘要顺序。
3. 如果桌面图和手机图某处冲突，以“视觉气质不变、交互适配手机”作为裁决原则。
4. 如果视觉稿中的某些数据当前 API 不提供，则保留视觉结构并用可得字段降级填充。

## 3. 首页必须照搬的部分

### 3.1 桌面首页必须高度还原

1. 左侧深色纵向导航的整体气质、宽度级别、图标列与品牌区布局。
2. 顶部全局栏的结构：搜索框、连接状态、刷新状态、主题/语言/通知/环境/用户区。
3. Hero 指标条的 6 卡布局、卡片高度、图标位置、主数字与小趋势线的关系。
4. 筛选条的双层结构：上层快捷筛选 chips，下层搜索与下拉筛选。
5. 中央主内容的双区排布：节点卡矩阵 + Global Overview / Insights 区。
6. 底部分析卡片排布：地图、流量趋势、区域延迟、事件环图、告警列表、系统概览。
7. 整体材质：深色玻璃、边框辉光、微噪点背景、卡片内发光和图表线条风格。

### 3.2 移动首页必须高度还原

1. 顶部品牌区、搜索、连接态、菜单按钮的相对位置。
2. 首屏 5 张状态卡的尺寸节奏与彩色边缘发光。
3. 中层快捷 tabs + 过滤器抽屉入口。
4. 主列表卡片的横向指标带、状态 badge、到期信息、底部 mini metrics。
5. 底部导航和右下角悬浮操作按钮的产品气质。

## 4. 节点详情页必须照搬的部分

### 4.1 桌面详情页必须高度还原

1. 左侧导航与桌面首页统一的产品壳子。
2. 顶部 breadcrumb、主标题、状态 badge、区域信息、操作按钮的布局关系。
3. 第一屏 6 张资源卡的尺寸、边框、图标和迷你趋势线样式。
4. 双大图表的并排结构与时间范围切换气质。
5. 下方异常卡片和资产账单卡片的二段式布局。
6. 右侧纵向信息侧栏的块级结构与信息层次。

### 4.2 移动详情页必须高度还原

1. 顶部品牌区、返回、通知、更多菜单的节奏。
2. Hero 卡片的整体比例、状态展示、核心标识字段排布。
3. 快捷操作按钮的四等分布局。
4. 资源卡片的两列网格和中间 tab 切换。
5. 主图表、次图表、告警卡、资产详情折叠区的层级顺序。

## 5. 允许动态适配的部分

这些部分可以按当前 Komari 公开数据能力做内容降级，但视觉结构仍然要尽量保留。

1. 首页 `Alerts`、`Incidents`、`Reports`、`Integrations`、`Billing` 等菜单项，可映射为当前真实可用 public routes 或占位入口，不强行伪造后台能力。
2. 首页 `Quick Insights`、`Latest Alerts`、`System Overview` 的数据来源，可先基于已有 live data、节点列表和 recent 数据做近似实现。
3. 首页 `Production / Staging / Reserved / GPU / High Traffic / Expiring Soon` 等 chips，允许先映射为 group、tag、price、traffic、expired_at 等可得字段。
4. 详情页右侧的 `Security & Ports`、`Recent Events`、`Ownership` 等信息，若当前 API 不提供，可做降级卡片或先隐藏具体字段。
5. 详情页 `Open in Admin`、`Compare`、`Share` 等按钮可以先做壳子和交互入口，再逐步补实功能。

## 6. 明确不允许的偏差

1. 不允许把视觉稿的深色高级 SaaS 气质做成普通开源后台风。
2. 不允许把桌面首页缩减成“只有节点列表”的实现。
3. 不允许因为现有数据有限，就把大卡片、信息层级和双区布局全部砍掉。
4. 不允许把移动端做成桌面布局的机械缩放版。
5. 不允许在实现过程中随意更换色相、圆角体系、边框亮度和图表线条风格。

## 7. 语言与文案要求

1. 首页和详情页必须同时支持简体中文和英文。
2. 首次访问默认显示简体中文。
3. 用户切换语言后，后续访问应保持原选择。
4. 中英文都要根据视觉稿控制文案长度，避免状态 badge、按钮、标题和指标标签破版。
5. 实现时优先保证中文观感，再验证英文等长替换后的布局稳定性。

## 8. 开发时的执行顺序

1. 先抽 token 和通用材质层，避免首页和详情页分别手写一套视觉。
2. 先做桌面首页骨架，再同步做移动首页。
3. 先做桌面详情页骨架，再同步做移动详情页。
4. 每完成一个页面，都必须同时做中英文和桌面/手机双回归。

## 9. 验收口径

如果后续让我按这份基准继续实现，默认以这 4 条作为验收口径：

1. 第一眼截图对比时，视觉语言、布局骨架和材质气质必须明显接近基准图。
2. 即使数据内容不同，卡片密度、层级、留白、图表样式也不能明显跑偏。
3. 中文默认显示，英文切换可用且不破版。
4. 桌面和移动端都必须像产品成品，而不是半成品适配稿。
